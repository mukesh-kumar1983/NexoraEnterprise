Decision

NexoraEnterprise will use a Tenant-centric multi-tenant architecture.

A Tenant represents a single customer/business/legal entity that subscribes to the platform.

The system will not introduce an Organization entity in the initial architecture.

Every business entity (company) is represented as a separate Tenant.

Why we chose this

1. Simpler Domain Model

Instead of:
Tenant
└── Organization
└── Employee

we have:
Tenant
└── Employee

2. Simpler Security

Every authenticated user belongs to exactly one Tenant.

JWT contains:

UserId
TenantId
Roles
Permissions

No OrganizationId.

3. Simpler Database

Every business table contains only:

TenantId

Examples:

Employees
Departments
Branches
Customers
Vendors
Projects
Assets
Payroll
Attendance
Leaves
Invoices

No OrganizationId.

4. Simpler Global Query Filters

Instead of:

< />c#

TenantId &&
OrganizationId

every filter becomes:

< />c#

TenantId

which keeps EF Core global filters and repositories much cleaner.

5. Simpler Authorization

Every request already knows:

Current User
↓

Current Tenant
↓

Current Roles
↓

Current Permissions

No organization switching.

6. Easier Module Development

Every future module automatically follows the same rule:

Entity
↓
Tenant

Examples:

Employee
Department
Branch
Payroll
Leave
CRM
Inventory
Finance
Projects
HR

Everything is tenant-owned.

Data Ownership

The model becomes:

Platform

├── Tenant A

│ ├── Branches
│ ├── Customers

│ └── ...
│
├── Tenant B
│ ├── Users
│ ├── Employees
│ └── ...
│
└── Tenant C
└── ...

Future Scalability

If, in the future, there is a genuine requirement to support a holding company that
manages multiple legal entities with shared users and consolidated reporting, we can
introduce a new higher-level concept without changing the core tenant isolation model.
Until such a requirement exists, we will avoid adding that complexity.

Development Principle Going Forward

From today onward, our guiding principle will be:

Every business entity belongs directly to a Tenant. There is no Organization layer unless a validated business requirement emerges in the future.

This aligns well with your objectives for NexoraEnterprise:

- Enterprise-grade but not over-engineered.
- Clean Architecture with a rich domain model.
- Consistent tenant isolation.
- Easier maintenance and onboarding.
- Faster feature development with less architectural overhead.

I also suggest one more rule for our collaboration:

- Before implementing any new module, we'll first validate its place in the domain model.
- If it fits our tenant-centric architecture, we proceed; if it challenges a core assumption,
- we'll document it in a new ADR before writing code.

That approach will help NexoraEnterprise grow in a disciplined way without accumulating
architectural debt.
